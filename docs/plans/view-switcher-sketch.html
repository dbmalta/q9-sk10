View Switcher — Visual Sketch & Decisions

Mode pills (Member / Admin) + node scope picker in the topbar. Only renders when the user has at least one admin scope. An admin scope can be any node in the org hierarchy (district, group, team, etc.). Full decision log from 40 Q&A below.

Scenarios

Scenario A — Admin mode, scope "All nodes"
Volunteer with admin rights at two nodes. In admin mode. No specific scope selected (aggregate view).
ScoutKeeper
Lists show records across every node the user has rights to, with a node label column. Per-row capability checks drive edit/delete visibility.
Scenario B — Admin mode, scope "District A"
Same user, filtered to one district. Blue accent stripe signals admin mode. Scope picker shows leaf + parent when deep in the tree.
ScoutKeeper
All scopable queries filter by org_closure subtree of node 4.
Scenario C — Member mode
Same user toggled into member mode. Green stripe, no scope picker. "My people" view only — profile + participation views.
ScoutKeeper
Shows: my profile, my upcoming events, my achievements, my children's records, my RSVPs, fees — always unscoped (includes secondary memberships).
Scenario D — Pure member (no admin scopes) OR pure admin (no member record)
Switcher hidden entirely. Pure member sees member-only chrome; admin-only user (e.g. district coordinator) sees admin chrome with scope picker but no mode pills.
ScoutKeeper
Scenario E — Admin mode on a non-scopable page (e.g. system settings)
Scope pill is hidden. Mode pills remain so the user can exit admin mode.
ScoutKeeper
Scenario F — Single-scope admin
User has admin rights at exactly one node. Scope picker hidden entirely; the node is implicit and shown via breadcrumbs/page headers.
ScoutKeeper

Decisions (40 questions resolved)

Core model

  • Terminology: "Admin" mode (fits volunteers). Scope is any node in the org hierarchy.
  • Scoping pattern: single active scope at a time, with "All nodes" as aggregate.
  • Scope inheritance: picker lists only explicit assignments; cascade via org_closure.
  • Security principle: toggle is UX; server-side permission checks are always authoritative.
  • Setup wizard: first admin assigned at root node — no super-admin bypass; closure table handles "everything".

UX rules

  • Non-scopable pages: hide scope pill; keep mode pills.
  • Member mode: never shows scope picker.
  • Single-scope user: hide scope picker entirely.
  • Admin-only user (no member record): hide mode pills entirely.
  • Default mode on login: remember last per-user; stored on users row.
  • Last-selected scope fallback: if invalid (deleted / revoked), fall back to single valid scope or "All nodes", with one-time flash message.
  • Scope picker label: leaf + parent (e.g. Group 1 › Patrol Blue); full path in dropdown.
  • Mobile layout: switcher moves into offcanvas sidebar header.
  • Landing page: / and brand link resolve to the correct dashboard for current mode.
  • Empty states: two-state — truly empty vs. scope-empty; latter offers "switch to All nodes".
  • Unsaved forms: confirm dialog on scope switch if form is dirty.
  • Keyboard shortcut: none in v1.
  • Member mode content: profile + participation (events, achievements, children, RSVPs, fees).

Data & semantics

  • "All nodes" with mixed rights: union of readable records; per-row capability checks drive write buttons.
  • Mixed roles across tree: per-row capability check based on that row's node.
  • Cross-node records: single owning node_id + visibility cascade (visible to ancestors and descendants of owning node).
  • Multi-node membership: primary node_id on member + optional secondary memberships via link table.
  • Registrations: applicant specifies target_node_id at registration; routed to admins at that node.
  • Bulk import: always requires explicit target node (pre-selected from active scope if available).
  • Out-of-scope record access: silent redirect to member mode if the record is one of their own; otherwise friendly error.

URLs & navigation

  • Scope in URL: ?scope=<id> or ?scope=all; session seeds default; URL wins when present.
  • Mode in URL: explicit ?mode=admin|member; middleware injects it into admin-generated links.
  • Multi-tab: URL is truth; session only seeds initial value.
  • Scope is global: active scope applies to every scopable module uniformly.

Module-specific behaviour

  • Search: scope-aware default with a "search all my nodes" toggle in results.
  • Audit: scope-filtered where possible; system-level entries visible only at "All nodes". Requires optional node_id on audit_log.
  • Reports: respect active scope.
  • iCal feed: ignores scope; returns everything the user can see (personal schedule).
  • Export: respects active scope.
  • Backup: ignores scope (must be complete for restore).
  • Impersonation (if planned): inherit target user's mode + scope.
  • Emails: outbound links embed mode+scope appropriate to the email's purpose.
  • Cron: no session scope; generated links embed mode+scope per Q13.

Security & integrity

  • Scope validation on submit: re-query role_assignment_scopes on every switch.
  • Invalid scope submission: flash error + audit log entry; no rate limiting in v1.
  • Audit entries: record view_mode and active_scope_node_id at time of action.
  • Session cache invalidation: bust scope_cache on any role-assignment write.

Non-functional

  • i18n keys: grouped under view.*.
  • Accessibility: standard ARIA labels + aria-live region announcing mode/scope changes.
  • Performance: no materialised descendant cache in session — rely on org_closure join at query time.
  • Testing: unit tests for ViewContextService + integration tests per scopable service + E2E for critical flows.
  • Rollout: no feature flag — ship it (pre-v1, no existing users).
  • Documentation: update CLAUDE.md + save memory entry after implementation lands.

Session shape

$_SESSION['view_mode']            = 'member' | 'admin';
$_SESSION['active_scope_node_id'] = int | null;   // null = "All nodes"
$_SESSION['scope_cache']          = [             // rebuilt on login / role change
    ['node_id' => 4,  'path' => 'District A',                 'leaf' => 'District A'],
    ['node_id' => 12, 'path' => 'District A › Group 1', 'leaf' => 'Group 1'],
];
// URL params ?mode= and ?scope= override session on a per-request basis.

DB additions

  • users.view_mode_last enum('admin','member')
  • users.scope_node_id_last int null
  • audit_log.node_id int null (for scope-filtered audit view)
  • audit_log.view_mode varchar(10) null (for forensic context)
  • New link table member_node_memberships for secondary memberships (primary stays on members.node_id)

Endpoints

  • POST /context/mode — body: mode=admin|member
  • POST /context/scope — body: node_id=<int> or node_id=all

Both CSRF-protected; both redirect to redirect_to. Both persist to users table.
